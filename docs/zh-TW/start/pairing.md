# 配對

「配對」是 OpenClaw 明確的 **擁有者核准** 步驟。
它用於兩個地方：

1. **私訊配對**（誰被允許與機器人對話）
2. **節點配對**（哪些裝置／節點被允許加入 Gateway 閘道器 網路）

安全背景：[Security](/gateway/security)

## 1) 私訊配對（入站聊天存取）

當頻道設定為私訊政策 `pairing` 時，未知的傳送者會收到一組短碼，且其訊息在你核准之前 **不會被處理**。

預設的私訊政策記載於：[Security](/gateway/security)

配對碼：

- 8 個字元，大寫，無易混淆字元（`0O1I`）。
- **1 小時後過期**。機器人只會在建立新的請求時送出配對訊息（大約每位傳送者每小時一次）。
- 進行中的私訊配對請求預設每個頻道最多 **3 筆**；在其中一筆過期或被核准之前，其餘請求會被忽略。

### 核准傳送者

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

支援的頻道：`telegram`、`whatsapp`、`signal`、`imessage`、`discord`、`slack`。

### 狀態存放位置

儲存在 `~/.openclaw/credentials/` 之下：

- 進行中的請求：`<channel>-pairing.json`
- 已核准的允許清單儲存區：`<channel>-allowFrom.json`

請將這些資料視為敏感資訊（它們用來控管對你的助理的存取）。

## 2) 節點裝置配對（iOS／Android／macOS／無頭節點）

節點以 **裝置** 的形式連線至 Gateway 閘道器，並使用 `role: node`。Gateway 閘道器
會建立一個必須被核准的裝置配對請求。

### 核准節點裝置

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

### 狀態存放位置

儲存在 `~/.openclaw/devices/` 之下：

- `pending.json`（短期存在；進行中的請求會過期）
- `paired.json`（已配對的裝置 + 權杖）

### 備註

- 舊版的 `node.pair.*` API（CLI：`openclaw nodes pending/approve`）是
  由 Gateway 閘道器 擁有的獨立配對儲存區。WS 節點仍然需要進行裝置配對。

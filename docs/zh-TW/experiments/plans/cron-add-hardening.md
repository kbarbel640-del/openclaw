# Cron Add 強化與結構描述對齊

## 背景

近期的 Gateway 閘道器 日誌顯示，出現多次因參數無效而導致的 `cron.add` 失敗（缺少 `sessionTarget`、`wakeMode`、`payload`，以及格式錯誤的 `schedule`）。這表示至少有一個用戶端（很可能是代理程式工具的呼叫路徑）正在傳送被包裝或僅部分指定的工作負載。另一方面，TypeScript、Gateway 閘道器 結構描述、CLI 旗標與 UI 表單型別中的 cron 提供者列舉值之間也出現了漂移，此外，UI 對於 `cron.status` 也存在不一致（期望 `jobCount`，但 Gateway 閘道器 回傳 `jobs`）。

## 目標

- 透過正規化常見的包裝負載並推斷缺失的 `kind` 欄位，停止 `cron.add` INVALID_REQUEST 的垃圾訊息。
- 在 Gateway 閘道器 結構描述、cron 型別、CLI 文件與 UI 表單之間對齊 cron 提供者清單。
- 讓代理程式 cron 工具的結構描述明確化，使 LLM 能產生正確的工作負載。
- 修正 Control UI 中 cron 狀態的工作數量顯示。
- 新增測試以涵蓋正規化與工具行為。

## 非目標

- 變更 cron 排程語意或工作執行行為。
- 新增新的排程種類或 cron 表達式解析。
- 除了必要的欄位修正之外，不對 cron 的 UI／UX 進行全面重構。

## 發現事項（目前的缺口）

- Gateway 閘道器 中的 `CronPayloadSchema` 排除了 `signal` 與 `imessage`，但 TS 型別包含它們。
- Control UI 的 CronStatus 期望 `jobCount`，但 Gateway 閘道器 回傳 `jobs`。
- 代理程式 cron 工具的結構描述允許任意的 `job` 物件，導致可形成格式錯誤的輸入。
- Gateway 閘道器 對 `cron.add` 進行嚴格驗證而未做正規化，因此被包裝的負載會失敗。

## 變更內容

- `cron.add` 與 `cron.update` 現在會正規化常見的包裝結構，並推斷缺失的 `kind` 欄位。
- 代理程式 cron 工具的結構描述已與 Gateway 閘道器 結構描述對齊，從而減少無效的負載。
- 提供者列舉值已在 Gateway 閘道器、CLI、UI 與 macOS 選擇器之間對齊。
- Control UI 使用 Gateway 閘道器 的 `jobs` 計數欄位來顯示狀態。

## 目前行為

- **正規化：** 被包裝的 `data`/`job` 負載會被解包；在安全的情況下會推斷 `schedule.kind` 與 `payload.kind`。
- **預設值：** 當缺失時，會為 `wakeMode` 與 `sessionTarget` 套用安全的預設值。
- **提供者：** Discord／Slack／Signal／iMessage 現在會在 CLI／UI 中一致地顯示。

請參閱 [Cron jobs](/automation/cron-jobs) 以了解正規化後的結構與範例。

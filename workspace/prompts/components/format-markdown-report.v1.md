# format-markdown-report.v1

Markdown 報告格式組件

## 格式定義
不是文檔，是能讓決策者在電梯裡讀完的行動指南。

## 核心原則
- **掃描友好**：3 秒看懂結構，30 秒抓住重點
- **行動導向**：每個段落都指向下一步
- **視覺層次**：用格式說話，不靠廢話
- **手機優先**：在任何螢幕上都賞心悅目

## 報告結構

### 1. 標題區塊
```markdown
# [動詞] + [核心主題]：[預期成果]
> 一句話執行摘要（最多 20 字）

**關鍵數字**：[最重要的 3 個數據]
**時間框架**：[明確的時間範圍]
**決策需求**：[需要什麼決定/資源]
```

### 2. TL;DR 區塊（太長不讀版）
```markdown
## 🎯 核心要點
1. **發現了什麼**：[一句話說明]
2. **這意味著什麼**：[商業影響]
3. **建議怎麼做**：[具體行動]
```

### 3. 背景脈絡
```markdown
## 📋 背景
### 為什麼現在
- 觸發點：[什麼事件/數據/需求]
- 時機性：[為什麼不能等]
- 關聯性：[與其他計劃的關係]

### 當前狀態
| 維度 | 現狀 | 目標 | 差距 |
|------|------|------|------|
| [關鍵指標1] | [數值] | [數值] | [%] |
| [關鍵指標2] | [數值] | [數值] | [%] |
```

### 4. 分析洞察
```markdown
## 🔍 關鍵洞察

### 洞察 1：[一句話標題]
**數據支撐**：
- 證據 A：[具體數據/事實]
- 證據 B：[具體數據/事實]

**深層含義**：
> [這告訴我們什麼？對業務的影響？]

### 洞察 2：[一句話標題]
[同樣結構...]
```

### 5. 方案設計
```markdown
## 💡 解決方案

### 方案概覽
```
[視覺化圖表/流程圖用 ASCII 或 mermaid]
```

### 實施路徑
#### Phase 1：[名稱]（Week 1-2）
- [ ] 行動項 1：[具體任務]
- [ ] 行動項 2：[具體任務]
- **產出**：[可交付成果]
- **成功標準**：[如何判斷完成]

#### Phase 2：[名稱]（Week 3-4）
[同樣結構...]
```

### 6. 資源需求
```markdown
## 💰 資源評估

### 投入
| 資源類型 | 數量 | 用途 | 優先級 |
|----------|------|------|---------| 
| 人力 | [人天] | [做什麼] | P0/P1/P2 |
| 預算 | [$] | [買什麼] | P0/P1/P2 |
| 時間 | [週] | [達成什麼] | - |

### 預期回報
- **短期（1個月）**：[具體成果]
- **中期（3個月）**：[業務影響]
- **長期（1年）**：[戰略價值]

### ROI 計算
> 投入：[X] → 產出：[Y] → 回報率：[Z%]
```

### 7. 風險管理
```markdown
## ⚠️ 風險與對策

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|--------|------|----------|
| [風險描述] | 高/中/低 | 高/中/低 | [具體對策] |
```

### 8. 行動呼籲
```markdown
## 🚀 下一步行動

### 立即需要（24小時內）
1. **決策**：[需要誰決定什麼]
2. **批准**：[需要什麼資源/權限]
3. **行動**：[第一個具體步驟]

### 本週目標
- [ ] [可完成的里程碑 1]
- [ ] [可完成的里程碑 2]
- [ ] [可完成的里程碑 3]

### 成功場景
> 💭 **30天後**：[描述成功的樣子]
```

## 格式化技巧

### 強調手法
- **粗體**：關鍵概念、重要數字
- *斜體*：引用、補充說明
- `代碼塊`：技術術語、系統名稱
- > 引用：核心洞察、關鍵結論

### 視覺元素
```markdown
# 使用 Emoji 增強可讀性
🎯 目標 | 🔍 分析 | 💡 想法 | ⚠️ 警告
✅ 完成 | 🚀 啟動 | 📊 數據 | 💰 預算

# 使用分隔線區分段落
---

# 使用列表組織信息
- 一級要點
  - 二級細節
    - 三級補充（盡量避免）

# 使用表格對比數據
| 對比項 | 選項 A | 選項 B |
|--------|--------|---------| 
| 優點 | ✓ 快速 | ✓ 穩定 |
| 缺點 | ✗ 成本高 | ✗ 速度慢 |
```

### 顏色運用（GitHub 風格）
```diff
+ 新增的好處/機會
- 移除的風險/成本
! 需要注意的要點
# 中性的說明信息
```

## 寫作指南

### DO ✅
1. **倒金字塔**：最重要的放最前面
2. **一段一點**：每段只說一件事
3. **數據說話**：用數字代替形容詞
4. **視覺優先**：能用圖表不用文字
5. **行動結尾**：每節都有下一步

### DON'T ❌
1. **廢話連篇**：「眾所周知」「毋庸置疑」
2. **被動語態**：「將被完成」→「我們將完成」
3. **模糊描述**：「很多」「一些」→ 具體數字
4. **純文字牆**：超過 5 行就要換個格式
5. **開放結尾**：「請大家討論」→ 明確建議

## 場景模板

### 專案提案
```markdown
# 啟動 [專案名]：[預期業務成果]
> 用 [時間] 和 [資源]，提升 [指標] 達 [數值]

## 🎯 核心要點
1. **機會**：[市場/用戶需求]
2. **方案**：[核心解決思路]  
3. **回報**：[預期ROI]

[... 按照完整結構展開 ...]
```

### 問題分析
```markdown
# 解決 [問題]：[影響範圍] 用戶受影響
> 根因分析完成，需要 [資源] 在 [時間] 內修復

## 🎯 核心要點
1. **症狀**：[用戶看到的現象]
2. **根因**：[技術層面的原因]
3. **方案**：[修復計劃]

[... 按照完整結構展開 ...]
```

### 成果彙報  
```markdown
# [專案名] 完成：達成 [核心指標] 提升 [百分比]
> [時間週期] 內完成既定目標，[下一步計劃]

## 🎯 核心要點
1. **達成**：[具體成果列表]
2. **學習**：[關鍵經驗教訓]
3. **展望**：[後續機會]

[... 按照完整結構展開 ...]
```

## 使用範例
```yaml
prompt: |
  {{format-markdown-report.v1}}
  
  請將以下信息整理成報告：
  - 網站載入速度慢，影響轉換率
  - 分析發現圖片太大，JS 檔案沒壓縮
  - 建議優化前端資源
```

## 預期輸出
```markdown
# 優化網站效能：提升轉換率 15%
> 2 週內將載入時間從 5 秒降至 2 秒

**關鍵數字**：載入時間 -60% | 轉換率 +15% | 投資 20 人天
**時間框架**：2025/01/20 - 2025/02/03
**決策需求**：批准 2 名前端工程師全職投入

## 🎯 核心要點
1. **發現了什麼**：首頁載入需 5.2 秒，導致 40% 用戶流失
2. **這意味著什麼**：每月損失約 $50K 潛在營收
3. **建議怎麼做**：立即啟動前端效能優化專案

## 📋 背景
### 為什麼現在
- 觸發點：轉換率連續 3 週下降 8%
- 時機性：競爭對手剛推出秒開的新網站
- 關聯性：配合 Q1 行銷活動，必須改善體驗

### 當前狀態
| 維度 | 現狀 | 目標 | 差距 |
|------|------|------|------|
| 頁面載入時間 | 5.2s | 2.0s | -61% |
| 跳出率 | 68% | 45% | -23% |
| 轉換率 | 1.8% | 2.1% | +17% |

[... 報告繼續 ...]
```

## 記住
> 好的報告不是寫出來的，是設計出來的。
> 讀者的時間比你的文采更重要。

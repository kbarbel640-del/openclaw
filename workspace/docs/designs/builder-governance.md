## 📋 一、問題定義

### 傳統薪資制度的痛點
1. **模糊性**：依靠主管主觀評價，缺乏透明標準
2. **階層性**：晉升 = 管人，不適合技術型/創作型團隊
3. **僵化性**：年資導向，無法反映實際貢獻
4. **分離性**：薪資與工作內容脫鉤，缺乏意義連結

### 目標場景
- AI 時代的知識工作者團隊
- 強調模組化協作的技術團隊
- 需要清晰成長路徑的新創組織
- 追求「責任 ↔ 報酬」對等的文化型組織

---

## 🎯 二、解決方案

### 核心理念：「語場貢獻制」

> 薪資從來不只是報酬，它是一種結構性的責任表述。

**設計哲學**：
- 薪資 = 你在系統中能承擔的重量
- 等級 = 可被信任的責任上限
- 經驗值 = 可觀測的貢獻累積

### 方案架構

```
┌────────────────────────────────────────┐
│           Builder Pathway              │
├────────────────────────────────────────┤
│  EXP 經驗值系統                         │
│  ├── 語場對齊模組 (222)                 │
│  ├── 技能建構模組 (333)                 │
│  └── 導師信任模組 (555)                 │
├────────────────────────────────────────┤
│  等級 × 薪資映射                        │
│  ├── Lv.0 = 29,012 (起點)              │
│  ├── 每 1111 EXP = +1 級 = +1111 元    │
│  └── Lv.85 = 123,456 (頂點：Thinker)   │
└────────────────────────────────────────┘
```

## 🧠 三、核心概念

### 3.1 經驗值（EXP）系統

| 行為類型 | EXP/次 | 等級影響上限 | 說明 |
|---------|--------|-------------|------|
| 語場日誌撰寫 | +10/日 | 2 級 | 每日工作紀錄與反思 |
| 對齊會議參與 | +10/次 | 2 級 | 晨會/晚會參與 |
| 週結反思 | +30/週 | 2 級 | 每週回顧總結 |
| 技能模組完成 | +333~1111 | 2 級/模組 | 學習並驗收通過 |
| 任務模組實作 | +666~999 | 3 級/任務 | 實際交付成果 |
| 教學支援 | +111/次 | 2 級 | 協助他人學習 |
| 導師評分 | +87/週 | 2 級 | 導師給予觀察回饋 |
| 信任授權（一次性）| +1111 | 5 級 | 最高治理者認定 |

### 3.3 三大設計原則

| 原則 | 說明 | 實踐方式 |
|-----|------|---------|
| **可觀測** | 每次晉升有對應紀錄 | GitHub/Notion/Google Sheets 留痕 |
| **可複製** | 任務設計為 SOP | 模組化、標準化、可教學 |
| **可進化** | 制度隨組織演進 | 每季制度迭代審議 |

## 📖 四、實施指南

### 4.1 啟動步驟

1. **定義起薪與級距**
   - 設定 Lv.0 起薪（建議有意義的數字）
   - 設定每級增幅（建議固定金額，易於理解）

2. **設計經驗值來源**
   - 日常行為（日誌、會議、反思）
   - 技能認證（學習模組完成）
   - 任務交付（實際產出）
   - 信任授權（治理者認定）

3. **建立紀錄平台**
   - 選擇工具（Notion/Google Sheets/GitHub）
   - 設計紀錄模板
   - 建立驗收流程

4. **設計治理架構**
   - Builder 本人：申報、紀錄
   - Mentor 導師：觀察、回饋
   - 模組負責人：驗收、審核
   - 治理者：信任授權、制度調整

### 4.2 治理節奏

| 週期 | 活動 | 目的 |
|-----|------|------|
| 每日 | 語場紀錄 | 累積 EXP |
| 每週 | 週結反思 | 對齊進度 |
| 每月 | Builder Retrospective | 回顧與調整 |
| 每季 | 制度升級會議 | 優化規則 |

### 4.3 特殊情境處理

**回溯補發（Backpay）**
- 適用：制度上線前已完成的貢獻
- 流程：Builder 整理 → 導師審核 → 治理者核發
- 上限：信任型補發最多影響 5 級

**爭議處理**
- 可提交補充紀錄
- 由第三方（Mentor/治理者）審議
- 建立判例供未來參考

## 🎭 五、適用場景

### ✅ 適合
- 技術型/創作型團隊（非傳統管理架構）
- 強調自主性的組織文化
- 需要透明成長路徑的新創
- AI 時代知識工作者團隊

### ⚠️ 需調整
- 傳統階層組織（需簡化等級系統）
- 人數 <5 人（可能過於複雜）
- 高度變動的短期專案（紀錄成本高）

### ❌ 不適合
- 純粹按件計酬的工作
- 強調服從的傳統企業
- 無法建立持續紀錄習慣的團隊

## 🔮 六、可重用模式

### 模式 1：數字美學起薪法
```
起薪設計為有意義的數字序列：
29,012 → 30,123 → 31,234 → ...
每次加薪都是「密碼」，增加儀式感
```

### 模式 2：模組化任務系統
```
任務 ≠ 事件，任務 = 模組
模組特性：
- 獨立運作
- 可重複使用
- 可在系統中流轉演化
```

### 模式 3：三位一體映射
```
薪資 × 結構責任 × 信任
三者同步對應，避免分離
```

### 模式 4：自我申報 + 多方驗證
```
Builder 主動申報 → Mentor 觀察確認 → 負責人驗收
避免單一評價者的主觀偏差
```

# 幣塔員工培育系統 PRD

**版本**：v1.0
**日期**：2025-07-11
**作者**：產品團隊
**狀態**：Draft

---

## 1. Vision

> 讓每位客服員工在對的時機，收到對的引導，螺旋式長成最好的自己。

不是培訓系統，是**成長作業系統**——用 AI agent 團隊記住每個人的位置，在自然時機滴灌養分。

---

## 2. Problem Statement

| 現狀 | 問題 |
|------|------|
| 校準報告只看「對不對」 | 缺少「為什麼錯」和「下次怎麼做」的閉環 |
| 7 個人用同一套標準 | 忽略個體差異，兔兔和小峻的瓶頸完全不同 |
| 回饋是批次的 | 錯過最佳學習時機（犯錯後 5 分鐘 vs 週報時） |
| 無極 bot 單一角色承載所有 | context 窗口被管理+校準+閒聊稀釋，品質下降 |
| 成長無法量化 | 老闆靠感覺判斷，員工看不到自己的進步 |

## 3. User Personas

### 3.1 員工（7 人）

| 維度 | 描述 |
|------|------|
| 日常 | 在 Telegram 1:1 群組跟客戶對話、發校準截圖 |
| 痛點 | 不知道自己哪裡弱、校準結果看完就忘 |
| 期望 | 被看見進步、知道下一步該練什麼 |

**個體差異示意**：
- **兔兔**：話術熟練但容易情緒化 → 需要「貴賓狗切換」訓練
- **小峻**：穩定但缺乏引流意識 → 需要「社交NB」思維啟發
- **QQ**：新人，基礎校準還在建立 → 需要高頻正向回饋
- **Z/子墨**、**茂**、**小周**、**葦葦**：各有不同成長軌跡

### 3.2 管理者（杜甫）

| 維度 | 描述 |
|------|------|
| 日常 | 看校準數據、處理升級問題、制定話術策略 |
| 痛點 | 要逐一追蹤 7 人進度，沒有全局視角 |
| 期望 | 一眼看到團隊狀態，系統自動處理 80% 的日常引導 |

## 4. Core Features

### F1: 個人成長檔案（Growth Profile）

每位員工一份持續更新的結構化檔案，記錄：
- 當前能力雷達圖（校準準確率、語氣掌握、奧客處理、引流轉化、情緒穩定度）
- 螺旋位置（Phase 1-5，見 §7）
- 近期成長趨勢（上升 / 平台期 / 回落）
- 待解鎖的下一個能力點

### F2: 即時微教練（Micro-Coaching）

**觸發時機**（滴灌原則）：
- 校準報告出來後 → 針對性回饋（不是分數，是「這題你選 B 是因為⋯對吧？其實可以⋯」）
- 連續 3 次同類錯誤 → 推送專項小練習
- 突破個人紀錄 → 正向強化（「這週奧客處理零失誤 🎉」）
- 平台期超過 7 天 → 換個角度刺激（故事、案例、挑戰）

**不觸發**：
- 員工正在忙碌對話中
- 已經收到當日回饋（每日上限 2 則主動推送）

### F3: 角色切換訓練場

基於現有四大角色，設計情境模擬：
- 給員工一個客戶截圖 → 「你會用哪個角色回？」
- 系統給出建議角色 + 示範話術
- 追蹤每人在各角色的熟練度

### F4: 團隊儀表板（Manager View）

杜甫專用，一頁看完：
- 7 人雷達圖疊加比較
- 本週校準趨勢
- 需要關注的人（回落 / 長期平台期）
- 系統自動建議：「建議本週重點關注 QQ 的奧客處理」

### F5: 成長里程碑

可見的進度系統：
- Phase 完成通知
- 能力徽章（「奧客剋星」「引流達人」）
- 月度成長摘要（自動生成）

## 5. Agent 分工架構

### 核心原則：Context 隔離

每個 agent 只看自己該看的，不混。

```
┌─────────────────────────────────────────────┐
│                  嫦娥 (管理層)                │
│  職責：打卡、排班、下班匯總、行政通知          │
│  Context：考勤數據、排班表                    │
│  不碰：校準內容、員工成長數據                  │
├─────────────────────────────────────────────┤
│              氣質美女 (標準客服)               │
│  職責：80% 日常話術示範、校準評分              │
│  Context：話術庫、校準標準、當次對話           │
│  不碰：員工個人成長檔案、管理決策              │
├─────────────────────────────────────────────┤
│              貴賓狗 (升級處理)                 │
│  職責：奧客場景示範、升級語氣訓練              │
│  Context：奧客案例庫、升級話術                │
│  不碰：日常校準、行政事務                     │
├─────────────────────────────────────────────┤
│              社交NB (引流成長)                 │
│  職責：社群貼文、引流話術、拉新策略            │
│  Context：社群數據、引流案例                  │
│  不碰：客服校準、內部管理                     │
├─────────────────────────────────────────────┤
│           🆕 螺旋教練 (Growth Coach)          │
│  職責：個人成長追蹤、微教練推送、里程碑管理     │
│  Context：員工 Growth Profile、校準歷史趨勢   │
│  不碰：具體話術內容、行政事務                  │
│  觸發：校準完成事件、時間觸發、趨勢變化        │
└─────────────────────────────────────────────┘
```

### 資料流

```
員工發截圖 → 氣質美女/貴賓狗 校準評分
                    ↓ (事件)
              螺旋教練 接收結果
                    ↓
            更新 Growth Profile
                    ↓
          判斷是否觸發微教練 ──→ 推送給員工
                    ↓
            彙整到儀表板 ──→ 杜甫
```

### 新增 Agent：螺旋教練

| 屬性 | 說明 |
|------|------|
| 身份 | 不直接跟員工對話的「幕後教練」 |
| 輸入 | 校準事件、時間信號、Growth Profile |
| 輸出 | 微教練訊息（透過無極 bot 發送）、儀表板數據 |
| 語氣 | 溫暖但精準，像好的主管而非冷冰冰的系統 |

## 7. Growth Loop（螺旋成長模型）

### Phase 定義

| Phase | 名稱 | 特徵 | 系統行為 |
|-------|------|------|----------|
| 1 | **紮根** | 校準準確率 < 60%，在學基礎 | 高頻正向回饋、簡單練習、容錯空間大 |
| 2 | **穩定** | 準確率 60-75%，基本功到位 | 開始引入角色切換、減少回饋頻率 |
| 3 | **拓展** | 準確率 > 75%，單一角色熟練 | 推送跨角色訓練、引流意識啟發 |
| 4 | **整合** | 多角色切換自如 | 實戰案例挑戰、開始帶新人任務 |
| 5 | **自驅** | 能自主判斷、輸出方法論 | 減少系統介入、邀請貢獻案例庫 |

### Phase 升降規則

```
升級條件：連續 14 天符合下一階段指標 + 無重大回落
降級條件：連續 7 天跌破當前階段底線
回落保護：降級後 3 天觀察期，不推送負面回饋
```

### 飛輪效應

```
校準練習 → 即時回饋 → 能力提升 → 解鎖新角色
    ↑                                    ↓
里程碑成就感 ← 儀表板可視化 ← 成長數據累積
```

## 8. 分期規劃

### MVP（Phase 1：4 週）

**目標**：跑通「校準 → 回饋 → 記錄」閉環

| 功能 | 說明 |
|------|------|
| ✅ Growth Profile 建檔 | 7 人基礎檔案，手動初始化 |
| ✅ 校準後自動回饋 | 校準報告 → 螺旋教練生成 1 句針對性回饋 |
| ✅ 技能分數自動更新 | 每次校準後更新 profile |
| ✅ 簡易儀表板 | `/dashboard` 指令看 7 人現狀 |
| ❌ 角色訓練場 | 先不做 |
| ❌ 里程碑系統 | 先不做 |

**驗證指標**：
- 員工是否讀回饋（已讀率 > 70%）
- 回饋是否有用（員工主動回覆率 > 20%）

### V1（Phase 2：8 週）

| 功能 | 說明 |
|------|------|
| ✅ 螺旋 Phase 自動判定 | 根據歷史數據自動分 Phase |
| ✅ 滴灌推送引擎 | 時機判斷 + 頻率控制 |
| ✅ 連續錯誤偵測 | 3 次同類 → 專項推送 |
| ✅ 角色切換練習 | 簡單版：給場景、選角色 |
| ✅ 里程碑 v1 | 徽章 + 紀錄突破通知 |

### V2（Phase 3：持續迭代）

| 功能 | 說明 |
|------|------|
| 團隊對比分析 | 找出團隊共性弱點 |
| 個人化學習路徑 | 根據 Phase + 弱項自動排課 |
| 老帶新機制 | Phase 4-5 員工參與新人指導 |
| 話術庫眾包 | 員工貢獻優秀案例回饋到話術庫 |

## 9. Success Metrics

### North Star Metric
**團隊平均校準準確率月環比提升**

### 分層指標

| 層級 | 指標 | MVP 目標 | V1 目標 |
|------|------|----------|---------|
| 健康 | 回饋已讀率 | > 70% | > 85% |
| 參與 | 員工主動回覆率 | > 20% | > 35% |
| 成長 | Phase 升級人數/月 | ≥ 1 人 | ≥ 2 人 |
| 效率 | 管理者手動介入次數 | baseline | -30% |
| 品質 | 校準準確率 MoM | +3% | +5% |
| 留存 | 員工滿意度（月度問卷） | — | > 4/5 |

### 反指標（要監控但不追求）

- 推送數量（多不等於好）
- Phase 升級速度（快不等於穩）

## 10. 風險與緩解

| 風險 | 影響 | 緩解 |
|------|------|------|
| 員工覺得被監控 | 抵觸使用 | 強調成長而非考核，語氣溫暖 |
| 回饋不精準導致信任下降 | 員工忽略推送 | MVP 先人工審核回饋品質 |
| Agent context 互相污染 | 回覆品質劣化 | 嚴格隔離 + 事件驅動而非共享記憶體 |
| 數據冷啟動 | Phase 判斷不準 | 杜甫手動設定初始 Phase |

## 11. 技術備註

- **儲存**：Growth Profile 以 JSON 檔案存於 `data/growth-profiles/` 目錄，MVP 不需資料庫
- **事件總線**：校準完成 → 寫入事件 → 螺旋教練消費，用簡單的檔案隊列或函數調用
- **推送通道**：所有員工訊息透過無極 bot 在 1:1 群組發送
- **Agent 隔離**：每個 agent 獨立 system prompt + 獨立 context window，不共享對話歷史
